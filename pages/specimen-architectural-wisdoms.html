Architectural Wisdoms

Four decisions that shaped the build system. Each one is a before/after of what changed and why. These are not guidelines — they are the system's skeleton.

From Multiple Files to God Script

decision-1 · PR #24 · session 011 · lesson: multiple-files-to-god-script
Before · PR #19
page.html ├── shell.js ← fetch #2 ├── vextreme.js ← fetch #3 ├── lang-fab.js ← fetch #4 ├── demo-fab.js ← fetch #5 └── scope bundles ← fetch #6–7 Total: 5–7 requests Latency: ~200ms Each fetch = a failure point
After · PR #24
page.html └── vextreme-{slug}.js ← fetch #2 ├── viewmodel (baked in) ├── EN strings (baked in) ├── fab-lang.js └── fab-demo.js Total: 1 request Latency: ~100ms EN strings: zero fetch
Why: Multiple files = multiple requests = multiple failure points. Each fetch is a round-trip that can time out, return stale data, or silently fail. Build-time assembly eliminates all of that for EN.
"The build script is the assembler. The browser receives the output."

From Magic Strings to Named Constants

decision-2 · PR #20 · session 010 · lesson: magic-strings-to-constants
Before · PR #19
window.VEX_STRING_CATEGORY = 'demo'; // 'demo' is a magic string // IDE can't navigate to its meaning // Typos fail silently // Change 'demo' everywhere = grep hunt
After · PR #20
window.VEX_STRING_CATEGORY = Category.DEMO; // Ctrl+click DEMO → definition // Definition explains what it means // Build validates against constants // Change 'demo' → change one file
Why: String values are irrelevant. Variable names are intentions. Category.DEMO is IDE-navigable: Ctrl+click reaches the definition, which explains the category's meaning, valid values, and constraints. A typo in a string fails silently. A typo in a constant name is a syntax error.
"The variable name is the intention. The string value is irrelevant."

From Hunting to Blueprint

decision-3 · PR #22 · session 011 · lesson: hunting-to-blueprint
Before · PR #21
To add a feature: 1. Read strings-compile.js 2. Read build-specimens.js 3. Read lang-fab.js 4. Read demo-fab.js 5. Infer the pattern 6. Hope you didn't miss anything Time: 30 minutes Errors: likely
After · PR #22
To add a feature: 1. Read blueprint.json 2. Add entry to config/features/ 3. Done Time: 5 minutes Errors: caught by validate-blueprint.js
Why: Adjacent code inference is error-prone. Each file shows only part of the contract. The blueprint makes the full contract explicit in one place — what exists, what's required, how to extend. Validation catches mistakes before they reach CI.
"The blueprint is the map. The code is the territory. Read the map first."

JSON Keys Are Strings — The Mitigation Is Validation

decision-4 · PR #24 · session 011 · lesson: json-keys-are-strings
The problem
// Named constants work in JavaScript: const cat = Category.DEMO; // ✅ // JSON has no symbols, no enums: { "category": "demo" } // ← raw string // If slug renames → index.json stale // If build fails → silent mismatch // No type error. No warning.
The mitigation
// check-key-alignment.js on every PR: nodes: 88 total, 88 in index ✅ arcs: 16 total, 16 in index ✅ // If stale: missingFromIndex: ["new-slug"] extraInIndex: ["old-slug"] // Non-blocking — informational. // Run build-index.js to fix.
Why: JSON is string-based. Keys are strings. This is unavoidable — it's a format constraint, not an architecture flaw. The response is validation, not elimination. A non-blocking CI check posts a key alignment report on every PR, catching drift before it causes confusion.
"JSON is string-based. Keys are strings. This is unavoidable. The mitigation is validation, not elimination."
