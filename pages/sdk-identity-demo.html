SDK Identity Demo

What the private resolver decided, and why.

A static, public-safe look at the private Vextreme SDK's identity resolver — a component whose general problem and boundary are documented in this repo's own public/private architecture split, but whose implementation is built and maintained privately as commercial-depth work.

What this demo shows

The private Vextreme SDK includes a resolver that takes a synthetic UI identity signal and a set of localization key candidates, and decides which candidate the signal actually refers to — even when the obvious match (an exact id) doesn't resolve the ambiguity on its own.

The input

A synthetic element identity signal, and two localization key candidates that happen to share the same final segment at different levels of nesting — the kind of ambiguity an id alone can't resolve:

"element": {
  "id": "hero-title",
  "context": "landing"
},
"keyCandidates": [
  { "key": "pages.demo-site.landing.section.1.hero-title" },
  { "key": "pages.demo-site.about.section.1.hero-title" }
]

The result

Resolver output

Status resolved
Confidence weak
Resolved key pages.demo-site.landing.section.1.hero-title
Reason trail
  • Multiple key candidates share a final segment matching the element id.
  • Key candidate scoped to element context "landing" narrowed an otherwise ambiguous set.
Warnings none

Both candidate keys end in hero-title, so the id alone matches both — genuinely ambiguous. The resolver doesn't guess between them; it narrows using the element's own context (landing) and only resolves once that narrows the set to exactly one candidate. If context hadn't resolved it, the honest output would have been status: "ambiguous", not a forced pick.

This is a static public demonstration artifact. The private Vextreme-SDK generated this sanitized resolver output from a synthetic fixture, and public Vextreme displays the result. No live connection exists between this page and the private SDK. Viewing this page does not grant access to the SDK, private code, secrets, or client data.

What this demo does not show

  • No private SDK source code.
  • No client data — every value on this page is invented.
  • No secrets, credentials, or tokens.
  • No live API — this page makes no network calls to anything private.
  • No authenticated user flow.
  • No production service — this is a controlled proof, not a live product.
  • No full localization engine — the resolver is one component, not the whole system.
