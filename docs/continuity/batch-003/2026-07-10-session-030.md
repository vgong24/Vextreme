### What was built and why

Victor's request had two parts: (1) review the two supplied business documents with actual
repository grounding rather than taking their claims at face value, and (2) build a
discoverable, reusable map of this project's recurring process types, since this is now the
second time a business-document-review pattern has occurred without a formal home.

Both parts turned out to depend on the same piece of missing infrastructure:
`cross-model-orchestration.md` treats business/stewardship content only as an *exit* from
the automated loop, never as a *defined* review workflow in its own right. Building
`business-document-review-protocol.md` first made the actual review (done in the companion
`Vextreme-SDK` PR, not this repo, per the routing rule the protocol itself establishes) a
defined process rather than an improvised one — and made the process map's second row real
rather than aspirational.

The review itself surfaced a concrete, load-bearing lesson: running `node lib/audit-nav.js`
live during the review found this repo's own `CLAUDE.md` claim ("31 of 39 pages are dead
ends") had already gone stale the same day it was written (Session 029's nav/FAB rollout
landed after that line was authored) — actual state is 33/39 navigable, with the remaining
6 confirmed as empty placeholders or an intentional dev fixture, not real dead content. That
finding is recorded in the review itself (companion `Vextreme-SDK` PR) and folded into this
protocol as a named standing principle: verify live, treat static claims — this repo's own
included — as provisional, not settled.

### Mistakes made

One, caught before this PR was opened: an initial pass pre-created `docs/continuity/
batch-004/` and flipped the registry's active row to it, reasoning from batch-003's README
text ("When Session 030 closes, this batch is full: create batch-004/...") as if it meant
"do this immediately." Running `npm run pr-ready` caught it —
`lib/check-map-bindings.js`'s own real validation flags an active batch directory with no
session files matching its declared range, and an empty `batch-004/` is exactly that.
Corrected by reverting the rollover: batch-003 stays active with session-030 as its
(now-final) real entry; `batch-004/` gets created at the actual start of Session 031, not
pre-emptively. Worth naming as a small process lesson: a "create X when Y" instruction in a
guide file describes the next session's first move, not this session's own closing move,
even when this session is the one that makes Y true.

(Separately, the stale nav-coverage figure described above is `CLAUDE.md`'s pre-existing
text going stale on its own timeline, not a mistake introduced this session — noted here
because it directly shaped this session's work, not because this session caused it.)
