# Session 030 - Page capability health as a shared projection

**Date:** 2026-07-10
**Branch:** `codex/screen-capability-health`

## Transition

Archives already exposed porting and language progress, while Terrain exposed
repository and content organization. Feature health remained split across
`audit-pages`, `audit-nav`, `audit-fab`, string manifests, Analysis Mode, and
screenshot files, so answering "which screen has what?" still required a grep
tour and human synthesis.

This session adds `lib/build-page-health.js` as a read-only join over those
existing truths. It writes `data/page-health.json`; it does not create a second
page registry or allow a projection to authorize runtime changes.

## Interface boundary

- Archives answers page health. It renders compact state/FAB/theme/analysis
  badges and aggregate counts beside the existing porting/language view.
- Terrain answers organizational health. Its content nodes receive a compact
  capability digest and derived status without adding graph nodes or edges.
- Theme control and visible theme styles are separate fields. A mounted widget
  is not proof that a page visibly supports light/dark presentation.
- Screenshot evidence means a convention-matching file exists. Freshness and
  capture automation are not claimed.

## Observed baseline

The first real projection sees 39 page files: 30 with a FAB delivery path, 10
with screenshot evidence, 9 with identity bindings, 4 sorted through the
curated node/arc registry, and 6 isolated from every audited navigation path.
Only one page currently satisfies every selected health criterion; 32 remain
visible as attention rather than being hidden or mislabeled as broken.

## Verification boundary

Focused source/generated tests passed after preserving Terrain's old
screenshot fallback for pure callers that do not supply page health. The local
browser remained blocked by the environment's existing localhost restriction,
so no rendered visual claim is made in this session. Browser verification is a
remaining review step, not silently treated as complete.

## Next safe action

Review the health criteria against the rendered Archives and Terrain views,
then use the generated gaps to scope runtime coverage work. Do not mass-enable
features from the projection alone.
