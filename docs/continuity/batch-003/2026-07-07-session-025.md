### Context on arrival

Victor shared three context-note-shaped documents Codex produced but hadn't merged
(`wip/registry-documentation-standard.md`, `wip/ui-identiy-registry-graph.md`,
`wip/localization-registry-graph.md`) plus a bilingual engineering dossier drafted as two
separate full HTML files (`wip/victor-dossier-en.html`, `wip/victor-dossier-zh.html`, PR #69,
`VXG-070626-codex-registry-foundation`), and asked for two things: (1) judge whether the three
registry docs were well communicated into the codebase already, or belonged archived somewhere,
and (2) use the two dossier HTMLs as a working proof-of-concept for merging a bilingual page
through the string-org system while also building reverse-tracing back to usages.

### What was reviewed

Read all three wip docs in full, then diffed them against what Codex actually built in PR #69
(`docs/architecture/15–17`, `data/registry/documentation-standard.json`,
`lib/check-registry-docs.js`). Verdict: well communicated, not a shallow wrapper — the built
version is mechanically enforced (a real health check, not just prose), honestly scoped (L6
not L7, explicit "don't rewrite the whole repo" instruction in the PR body matching the wip
docs' own migration philosophy), and anchored to real existing systems (both arch-16 and arch-17
carry a "Current Repo Fit" table mapping the abstract graph onto `data/nodes.json`,
`data/strings/source`, etc.). One correction offered: the *raw* prose docs' natural long-term
home is `docs/continuity/context-notes/` (which already has the exact registry/health-check/
promotion apparatus for "preserved reasoning, not yet accepted architecture"), not `wip/` —
`wip/` has zero lifecycle for `.md` files, only `.html` drafts and `.json` placeholders.

### What was built and why

**Read the two dossier HTMLs and found real value beyond a demo:** the two files are
structurally identical (same CSS, same DOM `id`/`data-legacy-id` pairs, same section order),
differing only in text — meaning the merge was a genuine data-i18n candidate, not a rewrite.
Also found a real content bug: the ZH file's localization-pipeline record was missing its
metric callout entirely (`<div></div>` where EN had `"6 mo → 2 wk"`).

**Before touching pipeline code, surfaced a real constraint rather than silently deciding it:**
adding `zh` for real collides with **td-006** — an explicit, pre-existing hard gate: *"Decision
must be made before adding a third language. Do not design the third language's pipeline on the
current per-scope-combo model."* Asked Victor twice via `AskUserQuestion` rather than picking
unilaterally: first, how far to take the POC at all (chose: resolve od-001 for real, not a
page-local demo); second, given td-006's actual prescription (arc-chunked bundles), whether to
do a full site migration, a scoped pilot, or override the gate (chose: **scoped pilot** — build
the real mechanism, exercise it on one arc only).

**Shipped the pilot, end to end:**
- `data/arcs-v2.json` gained an opt-in `"bundlingStrategy": "arc-chunked"` field. Only the new
  `victor_dossier` arc sets it; the other 16 arcs are completely unaffected.
- `lib/build-arc-bundles.js` (new) merges every opted-in arc's member scopes into one
  `data/strings/compiled/arcs/{arc}.{lang}.json` per language — the alternative td-006 named,
  built for real instead of just discussed.
- `lib/build-vextreme.js` emits `window.VEX_STRING_ARC_BUNDLE` for a piloted slug;
  `widgets/fab-lang.js` checks it before the per-scope fan-out (one fetch instead of N);
  `lib/build-sw.js` precaches the opted-in bundles, bounded to (opted-in arcs × languages).
  Also corrected a stale lattice claim found while touching `fab-lang.js`: its context said
  fetched strings were cached in localStorage keyed by scope-combo — reading the real source
  showed that never happens; only the language *preference* persists.
- Wired `lib/build-arc-bundles.js` into `.github/workflows/build-index.yml` for real (between
  strings-compile and build-index) — `lib/check-lattice-edges.js` caught that the first draft
  of this claimed CI wiring that didn't actually exist yet, and the fix was to build the wiring,
  not soften the claim.
- `lib/vex-config.js` gained `Language.ZH` — confirmed first that `strings-compile.js`/
  `strings-check.js` are already language-generic (langs derived from source file keys, not a
  hardcoded enum), so this was a naming/governance addition, not new plumbing.
- **Resolved od-001 and updated td-006** as recorded decisions rather than silently closing
  them: od-001 removed from `data/status/open-discussions.json` (question answered); td-006
  rewritten to record what shipped, what's still open (the other 16 arcs), and the corrected
  localStorage claim; a new "Arc-chunked bundling pilot" section added to
  `docs/architecture/06-i18n.md` naming the honest known limitation (`supportedLangs` is
  site-wide, so 🇨🇳 becomes selectable everywhere even though only one page has zh content —
  degrades safely via the existing missing-key-shows-EN fallback, not silently broken).

**Extracted and merged the dossier for real:**

  ZH metric bug during extraction.
- Curated the page for the first time: added a `data/nodes.json` entry (id 76) and the new

(orphaned data-i18n references, unused keys) in the same informational posture as
`check-key-alignment.js`. Confirmed zero orphan keys on the new dossier page.
